## 1. 项目背景与核心痛点

### 1.1 政策与时代背景

随着中共中央、国务院《关于全面深化新时代教师队伍建设改革的意见》及教育部《新时代基础教育强师计划》落地推进，"高素质、专业化、创新型"教师培养成为国家教育战略的核心。然而在 AI 大爆发的当下，行业普遍聚焦面向终端学生（To-Student）的辅导工具，**面向师范生（To-Teacher）训练**的智能化基础设施仍处于空白阶段。

### 1.2 师范生培养的微观痛点

师范生在象牙塔中积累了完备的教育学理论，但走上讲台时往往会经历巨大的认知落差。其中**"答疑"环节**——在课堂上、课后接住学生的临场提问、识别学生的认知盲区并给出适切的解释——是师范生最缺乏练习的能力之一：

1. **同伴扮演的"温室效应"**
   微格教学中由同班师范生扮演的"中小学生"，因背景相同、知识接近，往往无法提出真正的儿童式困惑。试讲变成"互相帮台"，根本暴露不出真实学生的认知陷阱。

2. **答疑场景比试讲更稀缺**
   多数师范生在实习前甚至**没有完整经历过一次"被学生连续追问"的体验**。教资面试中模拟答疑的题量极少，而真实课堂里，一句"老师我没听懂"才是最考验功底的场面。

3. **缺乏可重复的脚手架训练**
   优秀的答疑需要识别学生的迷思（Misconception）、命中其真实困惑、用合适粒度的"脚手架"逐步纠偏。这套能力只能在大量真实对话中习得，但成本高、门槛高、反馈慢。

### 1.3 用户画像

**李同学（大三，数学师范专业，备考教资）**：
> "我教案能写得很漂亮，但真到模拟答疑的环节就慌——学生说'老师我觉得 1/2 加 1/3 等于 2/5'，我下意识就说'不对，应该是 5/6'，可学生为什么会这么想我根本没琢磨过。我需要一个能反复跟我'磨'答疑的工具。"

## 2. 产品定义与目标用户

### 2.1 产品定义

**EchoClass** 定位为 **"AI 驱动的师范生 1v1 答疑陪练系统"**：

> 师范生上传教案，系统基于学段认知特征 + 学科迷思概念库，**生成一个会问真实问题的虚拟学生**，并与师范生展开多轮 1v1 答疑对话。学生在被讲懂时主动宣告"懂了"，否则会按其错误前提继续追问、表达困惑、甚至跑题。

### 2.2 核心交互闭环

```
[1] 上传教案 (PDF/MD/TXT)
       │  pymupdf4llm 解析 + LLM 抽取 LessonMeta
       ▼
[2] 选学段 + 选虚拟学生人设
       │  6 学段 × 3 学生 = 18 种典型组合
       ▼
[3] 学生根据教案"主动提问"（StudentAgent.generate_questions）
       │  宽生成 + self-check + 类别多样性筛选
       │  category: clarify_concept / challenge_example / extend_topic / off_topic / stuck_misconception
       ▼
[4] 师范生选一个问题进入 1v1 答疑对话
       │  StudentAgent.respond_in_dialog 多轮回应（流式渲染）
       │  学生按人设说话（带犹豫、口头禅、孩子腔）
       │  含错误前提的问题不轻易"懂"，要被讲到点子上才行
       ▼
[5] 学生在末尾输出 [懂了] 标记 → 触发 self_resolved 结束本次答疑
       │  也可以由师范生 /resolve 主动标记或 /abandon 放弃
       ▼
[6] 进入下一个问题 / 切换学生 / 结束 session
```

### 2.3 与"虚拟课堂试讲"的关键区别

EchoClass 在 M1 阶段做了重要的产品转型：从"多 Agent 虚拟课堂试讲"收敛到"1v1 答疑陪练"。理由如下：

| 维度 | 多 Agent 虚拟课堂（已弃） | 1v1 答疑陪练（当前） |
|---|---|---|
| 训练焦点 | 教学设计、课堂管理、互动节奏综合考核 | **聚焦答疑环节的认知诊断与脚手架引导能力** |
| 体验复杂度 | 多学生并发、Director 调度、抢答打断 | 单线程对话，体验简单可控 |
| LLM 成本 | 多 Agent 同时上下文，token 消耗高 | 单 Agent 多轮，单次 session 成本可控 |
| 技术风险 | 多 Agent 调度、事件流、抢话死锁 | 复杂度收敛，质量更可控 |
| 教育价值密度 | 分散在 4 维度的浅层 | **集中、深、可量化**——单题答疑的成败明显 |

**1v1 答疑陪练**是更"小而美"且更可控的切入点，符合赛事 4-8 周开发周期的现实约束。

### 2.4 目标用户

- **主用户**：师范专业本科生（大三大四）、教资备考群体、新入职 1-2 年青年教师
- **决策方**：高校师范学院、各级教师进修学校、教培机构教师培训部
- **辅助用户**：教研员（用 EchoClass 沉淀的语料做教研）

## 3. 核心教育学理论与机制

EchoClass 的差异化护城河在于**大模型 Prompt 工程与经典教育学理论的深度嵌套**。

### 3.1 还原儿童认知边界：皮亚杰认知发展阶段理论

通用 LLM 默认具备成人推理能力，让其扮演中小学生时会"超模"——小学一年级开口讲集合论。我们在底层用**学段共性特征库（StageProfile）**强行注入皮亚杰认知约束：

```
6 档学段：p_lower / p_middle / p_upper / j_lower / j_upper / h
每档定义：piaget_stage / thinking_style / language_style / typical_expressions
```

例如 `p_middle`（小学中年级，9-10 岁，**具体运算阶段**）会被注入：
> "你必须依赖具象事物理解概念。看到符号化的代数式会困惑，无法做抽象假设性推理。语言简短直白、爱用'比如说'、'像'这样的表达。"

### 3.2 错得真实的灵魂：学科迷思概念库（Misconceptions）

真正的答疑训练价值在于让师范生"排雷"。我们结构化整理了 **21 份学科 × 学段的迷思概念库**（覆盖小初高数学/物理/化学/生物/语文/英语/历史/地理/政治）。每条迷思包含：
- `id` / `name` / `subject` / `stage_id`
- `key_points`（关联的教学重点）
- `typical_error`（学生最常出现的错误说法）
- `intervention_hint`（教师纠正的正确路径）

**触发逻辑案例**——小学三年级《分数的初步认识》：
- 系统从迷思库匹配到迷思 `math.p_middle.fraction.equal_parts`：
  > "把'分一半'等同于'平均分一半'，不区分是否相等"
- 生成的学生提问会自然带入这个错误前提：
  > "老师，我把苹果切两块，一大一小，那大的那块是不是也算 1/2 呀？"
- 在 1v1 对话中，**学生不会轻易接受老师含糊的更正**，必须老师明确指出"必须平均分"才会输出 `[懂了]`。

这就把师范生**逼到必须练就识别迷思 + 命中迷思 + 纠正迷思的脚手架能力**。

### 3.3 维果茨基最近发展区与脚手架（Scaffolding）

学生 Agent 的"懂"不是一蹴而就的：

- **第一次老师讲解**：学生表达"听到了"+ 提出新疑问或部分听不懂
- **多轮追问**：按错误前提继续表达困惑，不会过早"懂"
- **真懂触发**：只有当老师讲解**真正命中错误前提**时，才输出 `[懂了]` 标记

这一行为模式由 prompt（`student_chat.j2`）显式约束，**强制师范生练习"分级解释 + 命中靶心"的脚手架能力**，而不是"我已经讲了，学生不懂是学生的问题"。

### 3.4 双层人设建模（学段共性 × 个体差异）

- **学段共性层**（StageProfile）：约束认知上限，避免 LLM 超模
- **个体差异层**（Persona）：18 个人设覆盖优秀/中等/薄弱 × 不同性格（活泼/内向/走神/挑战权威/焦虑/...）

师范生在不同学段 + 不同人设组合下训练，习得的不是"一种解释方式"，而是**"针对不同认知水平、不同性格学生的差异化答疑策略"**。

## 4. 技术架构与关键实现

### 4.1 技术栈总览

| 层 | 技术 | 选型考量 |
|---|---|---|
| **前端** | Next.js 14 (App Router) · TypeScript · TailwindCSS · shadcn/ui | SSR 优化首屏；与 Vercel AI SDK 配合做流式渲染 |
| **流式渲染** | WebSocket（JSON Lines）+ holdback 缓冲 | 学生 `[懂了]` 标记不能漏发到前端，要在末尾保留 holdback 字符做剥离 |
| **后端 API** | FastAPI · Pydantic v2 · uv | 异步非阻塞、统一 `ApiResponse` 包络（code/message/data/request_id） |
| **LLM 接入** | OpenAI 兼容接口（ChatECNU ecnu-max 默认） | 国产合规；可切 DeepSeek / Qwen 等 |
| **RAG** | Chroma + pymupdf4llm | 教案 500 token 切片向量化，支持迷思与教案重点联合检索 |
| **Agent 编排** | StudentAgent（generate_questions + respond_in_dialog + stream_in_dialog） | 单 Agent 二阶段（宽生成 + self-check）保障问题质量 |
| **Few-shot 范例** | 6 学段 ask/chat 范例集合 | 按人设 effective_level + tag 自动挑选 2 个 ask + 1 个 chat |
| **持久化** | SQLite（M2 计划） | 会话/对话历史/问题记录 |

### 4.2 学生提问的"宽生成 + self-check + 多样性筛选"管线

```
StudentAgent.generate_questions(lesson_meta, count=3)
├── [1] 宽生成阶段：LLM 调用 student_ask.j2
│        生成 count + overshoot 个候选问题（含 category / difficulty / linked_*）
├── [2] 解析校验：构造 list[StudentQuestion]
├── [3] self-check 阶段：LLM 调用 student_check.j2
│        二次自评每条问题（人设贴合度 40 + 教育价值 30 + 教案相关性 30）
│        score < 60 或 keep=false 的剔除
└── [4] 多样性筛选：按 self_score 降序 + category 多样性贪心，取 top count
```

宽生成 + self-check 是单 Agent 提升输出质量的关键模式，避免 "1 次 LLM 输出 = 1 次结果"的脆弱性。

### 4.3 1v1 对话的流式 `[懂了]` holdback

学生末尾的 `[懂了]` 标记**不能流式推到前端**——否则用户会看到"学生说完话后突然冒出 `[懂了]` 三个字"，破坏体验。

我们在 `StudentAgent.stream_in_dialog` 中实现了 **HOLDBACK 缓冲机制**：
- LLM 增量推送时，永远保留**末尾 16 个字符**不发出
- 流结束后，用 `_parse_dialog_reply` 处理完整文本，剥离 `[懂了]`
- 把剩余可见尾部补一个 `delta` 事件 + 最终 `final` 事件

这套机制保证前端看到的是"学生自然说完话 → 后端给出 `self_resolved=true` 信号"。

### 4.4 RAG 的双索引：教案 + 迷思

```
[教案索引]  pymupdf4llm 解析 → LLM 抽取 LessonMeta → Chroma 向量切片
[迷思索引]  21 份 JSON 静态加载 → 按 (subject, stage_id, key_points) 匹配
```

学生在生成问题时，prompt 同时注入：
- 教案重点 / 难点（从 LessonMeta）
- 当前学段 + 学科 + 教学重点匹配到的 3 条以内迷思

学生提到 `category=stuck_misconception` 的问题时，会自然带入匹配到的迷思错误前提。

## 5. 竞品分析与差异化壁垒

| 维度 | EchoClass | 通用 C 端大模型（Kimi/ChatGPT） | 国外教师培训产品（TeachFX 等） |
|---|---|---|---|
| **角色扮演真实度** | 高保真：学段认知 + 18 人设 + 21 份迷思库三重约束 | 单线扁平，需用户自己写数千字 prompt | 较强但本土化弱 |
| **理论嵌套深度** | 皮亚杰 + 维果茨基 + 学科迷思已写进 prompt 与数据层 | 通用回答，少有"符合学情的真实错误" | 基于欧美教育学体系 |
| **本土化适配** | 完全适配国内教资考纲 + 国产合规 LLM | 无定向 | 严重水土不服，数据出境风险 |
| **使用门槛** | 上传教案即可，分钟级上手 | 极高，要求 prompt 工程能力 | 主要面向学区集采 |
| **训练颗粒度** | **聚焦答疑场景，单题深度训练** | 通用对话 | 整堂课分析 |

### 护城河

1. **数据壁垒**：21 份学科 × 学段迷思库 + 6 学段 ask/chat few-shot 范例，是同行短期无法爬取的核心资产
2. **教育学嵌套**：学段共性 × 人设 × 迷思的三重约束模板已沉淀到 prompt 层，独立于基座模型
3. **聚焦答疑的产品定义**：避开了"通用试讲产品"的红海，做单一场景的深度

## 6. 商业化路径

采取 **"B 端机构定基调 + C 端教资考生扩规模"** 的混合策略。

### 6.1 B 端：高校师范学院 / 教师进修学校

- **机构 SaaS 订阅**：按账号数阶梯定价（约 5-20 万元/年）
  - 提供导师管理后台：批量布置答疑任务、查看每个师范生的练习时长 / 命中迷思率 / 平均轮次
- **私有化部署**：对数据隐私要求极高的公立机构按需提供本地化算力方案（约数十万元一次性 + 年维保）

### 6.2 C 端：师范生 / 教资备考群体

- **Freemium**：每日 15 分钟免费练习
- **Pro 会员**（29 元/月 或 128 元/半年）：
  - 不限时练习
  - 答疑历史导出 / 个人成长曲线
  - 教资备考冲刺期（每年 3 月、11 月）的高频考点专题包

### 6.3 GTM 节奏

- **M0-M1（种子打磨）**：在华东师大封闭公测，招募 30-50 名核心用户
- **M2-M3（内容裂变）**：小红书 / B 站 / 抖音教资备考圈层种草
- **M4+（B 端攻坚）**：携 C 端真实数据切入师范院校采购

## 7. 数据合规与伦理

1. **大模型幻觉的教育严谨性**
   - 控制：报告 / 反馈环节强制附"原对话片段引用"做事实溯源；学科 / 计算环节可外挂代码解释器二次校验
2. **生成式 AI 服务备案**
   - 控制：底层 LLM 选用已通过国家网信办《生成式人工智能服务管理暂行办法》备案的国产基座（ChatECNU / DeepSeek / Qwen）
3. **AI 评价的过度信赖偏差**
   - 控制：所有 AI 反馈页面强制显示免责说明 —— "AI 反馈仅作为辅助参考，不替代人类导师的最终评价"
4. **训练数据隐私**
   - 控制：私有化部署方案承诺核心对话数据不出本地；公网版本明示用户数据使用范围与匿名化策略

## 附录 A · 关键术语

| 术语 | 含义 |
|---|---|
| **StudentAgent** | 单学生虚拟 Agent，提供生成提问、对话回应、流式回应三个能力 |
| **StageProfile** | 学段共性认知特征（皮亚杰阶段、思维方式、语言风格） |
| **Persona** | 个体学生人设，18 字段含性格、口头禅、迷思倾向等 |
| **Misconception** | 结构化的学科迷思概念条目，含 typical_error / intervention_hint |
| **LessonMeta** | 教案抽取结果（subject / grade / topic / objectives / key_points / difficult_points） |
| **StudentQuestion** | 学生提出的结构化问题（含 category / difficulty / linked_key_point / linked_misconception_id / rationale / self_score） |
| **DialogReplyResult** | 1v1 对话单轮回应（含 content / self_resolved 标志 / raw 原文） |
| **`[懂了]`** | 学生在对话末尾输出的自宣称解决标记，触发 session 该题结束 |
| **Stage 共性 × Persona 个体** | 双层建模：避免 LLM 超模 + 注入个体差异 |
