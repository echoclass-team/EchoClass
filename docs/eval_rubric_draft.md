> **状态**：v0.5 草案 — 用于 M2 内部讨论与 M3 评估 Agent prompt 设计的种子文档
> **作者**：C-Prod
> **最近更新**：2026-04-26
> **关联 Issue**：#73（M2 内容工作）

---

## 0. 文档定位

本文档是 EchoClass 1v1 答疑陪练系统的**评估 Rubric 草案 v0.5**，回答一个核心问题：

> **当一个师范生用我们的系统跟一节课的虚拟学生答疑完之后，我们怎么客观地说"他答得好不好"？**

v0.5 的目标不是「最终评分系统」，而是给 M3 阶段要落地的"评估 Agent"提供一份**可用的、有锚点描述的、能直接拿来跑试评分**的种子规则集。

本文档的下游消费者：
- **M3 评估 Agent**：直接把第 5 节的 LLM-Judge prompt 框架接入 `LLMClient.chat`，把第 3 节的指标作为输出 schema
- **C-Prod 后续迭代**：依据第 6 节试评分暴露的 rubric 不可操作之处，迭代到 v1.0
- **A-Agent**：阅读第 2 节的数据来源约定，决定 `QASession.summary()` / `dialog.history` 的字段是否需补充
- **B-Full**：阅读第 4 节的总分计算公式，决定反馈页可视化的指标卡片该长什么样

---

## 1. 设计原则

### 1.1 三个不做（避免 v0.5 走偏）

- **不做绝对分数**：5 维度各自独立打分，不强行加权汇总成一个 "100 分" 的笼统数。师范生培养场景里，绝对分易被滥用为 KPI；维度独立 + 改进建议更有教研价值
- **不做"对话好不好听"的主观打分**：优先用结构化字段（`covered_key_points` / `broken_misconception_ids` / `resolution_sources`）+ 客观计数（轮次数、追问数）；无法结构化的能力（如脚手架是否分级）才下放到 LLM-Judge，且必须给具体行为锚点
- **不做覆盖所有的角色评估**：本 rubric 只评**师范生（即操作者）**的答疑表现，不评虚拟学生说话像不像（学生像不像由 A-Agent 的 prompt 工程负责）

### 1.2 三个必做（v0.5 的硬要求）

- **每个指标必须有 0-4 五档描述性 anchor**：不能只写"分数 0-4"，要描述"4 分长什么样、0 分长什么样"，让两个人评同一段对话能得出相近结果
- **每个指标必须标注数据来源**：明确是从 `QASession.summary` / `dialog.history` 读，还是要 LLM-Judge 提取
- **必须有可跑的试评分样本**：第 6 节给出 3 个 mock 对话 + 试评分，用于暴露 rubric 在真实对话上的不可操作之处

## 2. 数据来源与三层取数

为避免每个指标都要 LLM-Judge 调用导致评估成本爆炸，rubric 把指标按取数难度分三层：

### 2.1 第一层 · 直接量化（来自 `QASession.summary()`，零额外调用）

A-Agent 已实现 `services/qa_session.py::QASession.summary()`，返回字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `total_questions` | int | 本节课产生的总问题数（dialogs 数） |
| `resolved` | int | status=resolved 的对话数 |
| `abandoned` | int | status=abandoned 的对话数 |
| `pending` | int | 还在 pending 队列的问题数 |
| `active` | int | 进行中的对话数 |
| `covered_key_points` | list[str] | 已 resolved 对话覆盖的 lesson key_points（去重） |
| `broken_misconception_ids` | list[str] | 已 resolved 对话破除的迷思 id（去重） |
| `resolution_sources` | dict | `{self_resolve, teacher_marked, abandoned}` 三种解决方式计数 |

> **C-Prod 取数**：直接序列化 JSON，无需任何额外 LLM 调用。

### 2.2 第二层 · 派生量化（来自 `dialog.history` + 教案 / 迷思库元数据，本地计算）

不在 `summary()` 直接给出，但可以通过遍历 `session.dialogs` 与外部元数据本地计算：

| 派生指标 | 计算方式 | 用于哪个维度 |
|---|---|---|
| 平均教师轮次 | `mean(len([m for m in d.history if m.role=="teacher"]) for d in resolved_dialogs)` | 维度 1 / 3 |
| 学生主动追问次数 | 学生消息中含 `?` 或 `？` 的数量 - 1（首问不算追问） | 维度 5 |
| 候选迷思命中率 | `len(broken_misconception_ids) / len(match_misconceptions(...).result)` | 维度 1 |
| 难点覆盖率 | `len(covered_key_points ∩ lesson.difficult_points) / len(lesson.difficult_points)` | 维度 2 |
| 问题分类多样性 | `len({d.question.category for d in resolved_dialogs})` 取值范围 1-5 | 维度 2 |
| 对话长度分布 | `[len(d.history) for d in dialogs]` 的均值 / 中位数 | 维度 3 |

> **C-Prod 取数**：v0.5 阶段先在 rubric 描述里说清如何算；M3 评估 Agent 落地时由 A-Agent 把这些算子封装到 `services/eval_metrics.py`（建议命名）

### 2.3 第三层 · LLM-as-a-Judge（需调用 LLM）

只有以下情形交给 LLM-Judge：
- 涉及"是否击中错误前提"、"解释分级是否适配人设"、"脚手架质量"等人类直觉判断
- 单次评估只调一次 LLM-Judge（一次性传入完整对话 history + 元数据，输出 5 维度结构化 JSON）

为压缩成本与方差，v0.5 约束：
- LLM-Judge **只调用一次**完成全部第三层指标的评估
- 输出必须是结构化 JSON（schema 见第 5 节）
- 评估模型用与生成不同的模型（例如生成用 `ecnu-max`，评估用 `deepseek-chat` 或 `qwen-max`），减少同模型自评偏差

## 3. 五维评估指标

下文每个维度形如：

```
### 3.x 维度 N：维度名（缩写：xxx）

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| ... | ... | L1/L2/L3 | ... | ... | ... |

#### 0-4 描述性 anchor（详细版）
...
```

`L1` = 第一层（summary 直接读）；`L2` = 第二层（派生量化）；`L3` = 第三层（LLM-Judge）。

### 3.1 维度 1：迷思破除（缩写：MR · Misconception Resolution）

> **核心问题**：师范生有没有真的把这节课该破的迷思破掉？

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| MR-1 | 迷思命中率 | L2 | <20% 候选迷思被破 | 40-60% | ≥80% |
| MR-2 | 迷思破除完成度 | L1+L2 | stuck_misconception 类问题 0% 被解 | 50% 被解 | ≥90% 被解 |
| MR-3 | 迷思干预精准度 | L3 | 老师从未识别错误前提，只重复对错判断 | 部分对话识别但解释跑偏 | 每次都精准命中错误前提并给出反例 |
| MR-4 | 平均破除轮次 | L2 | 每条迷思 ≥6 教师轮才破除 | 4-5 轮 | ≤3 轮且学生有复述 |

#### MR-1：迷思命中率（自动）

- 公式：`len(broken_misconception_ids) / len(match_misconceptions(subject, stage_id, key_points, ...))`
- 即"本节课迷思库匹配了 N 个候选迷思，最终 K 个被对话破除掉了"
- **0 分**（< 20%）：基本没碰本课的典型迷思
- **1 分**（20-40%）：偶有命中，多数迷思仍漂浮
- **2 分**（40-60%）：过半数候选迷思被覆盖
- **3 分**（60-80%）：只剩长尾几条迷思未触及
- **4 分**（≥ 80%）：本课该破的迷思基本都破了

#### MR-2：迷思破除完成度（自动）

- 公式：`len({d for d in resolved_dialogs if d.question.category=="stuck_misconception"}) / len({d for d in dialogs if d.question.category=="stuck_misconception"})`
- 即"卡迷思的问题里，老师真把它讲通了的占比"
- **0 分**：所有 stuck_misconception 问题最终 abandoned
- **2 分**：约一半被讲通
- **4 分**：≥ 90% 被讲通且 self_resolve / teacher_marked 比例健康（见 RR-2）

#### MR-3：迷思干预精准度（LLM-Judge）

- LLM-Judge 拿到对话 history + 学生迷思的 `description / typical_error / cause`，判断老师的解释是否：
  1. 在学生第一次表达迷思时**识别**到这是迷思而非随机错（不是只说"你错了再算一遍"）
  2. **明确指出错误前提**（例如学生说"力越大物体跑越快"，老师不能只说"不对"，要说出"你把力和速度混了"）
  3. 给出**反例或类比**让学生自己看到矛盾，而不是只灌输正确结论

- **0 分**：从未识别错误前提，只判对错
- **1 分**：偶有识别但表达模糊，学生听完仍困惑
- **2 分**：在 ≥1 个对话中精准识别并给出解释
- **3 分**：多数 stuck_misconception 对话都精准识别
- **4 分**：每次精准识别 + 反例 + 类比三件套俱全

#### MR-4：平均破除轮次（自动）

- 公式：`mean(教师消息数量 for d in resolved_dialogs if d.question.category=="stuck_misconception")`
- 越短越好（迷思一旦被精准命中，3 轮内即可破除；超过 6 轮多半是反复绕远）
- 但要警惕"过短=直接告知答案不脚手架"，所以 MR-4 必须与 TQ-3（脚手架质量）联读

| 4 分 | 3 分 | 2 分 | 1 分 | 0 分 |
|---|---|---|---|---|
| ≤3 轮且学生有复述 | 3-4 轮 | 4-5 轮 | 5-6 轮 | ≥6 轮 或 ≤2 轮（疑似直接告知） |

### 3.2 维度 2：重点覆盖（缩写：KC · Key-points Coverage）

> **核心问题**：这节课的教学目标，被多少借由学生提问 / 师生对话覆盖到了？

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| KC-1 | 重点覆盖率 | L1 | <25% | 50-75% | ≥90% |
| KC-2 | 难点覆盖率 | L2 | 0% | 50% | 100% |
| KC-3 | 问题分类多样性 | L2 | 全 1 类 | 3 类 | 5 类全有 |
| KC-4 | 跨学生覆盖 | L2 | 只 1 个学生提问 | 3 个学生 | ≥5 个学生 |

#### KC-1：重点覆盖率（自动）

- 公式：`len(covered_key_points) / len(lesson.key_points)`
- **0 分**（<25%）：仅触及边角，主线知识点几乎未涉及
- **2 分**（50-75%）：主要重点都被某轮对话覆盖
- **4 分**（≥90%）：教案重点几乎全覆盖

#### KC-2：难点覆盖率（派生）

- 公式：`len(covered_key_points ∩ lesson.difficult_points) / len(lesson.difficult_points)`
- 难点是教案里更难讲的部分，与一般重点分别评分

#### KC-3：问题分类多样性（派生）

- `category` 集合（共 5 类）：
  - `clarify_concept` — 学生求澄清概念
  - `challenge_example` — 学生挑战例子的边界
  - `extend_topic` — 学生延伸到课外
  - `off_topic` — 学生跑题（适度跑题反而真实）
  - `stuck_misconception` — 学生卡在迷思上

- **0 分**：只 1 类（多半是 clarify_concept 一统天下）
- **2 分**：3 类
- **4 分**：5 类全有（说明老师能驾驭各种问题类型）

#### KC-4：跨学生覆盖（派生）

- 在多学生 1v1 答疑场景下，本课至少有几个不同学生（不同 persona）参与了 resolved 对话
- **0 分**：只 1 个学生提问，其他人静默
- **2 分**：3 个学生
- **4 分**：≥5 个学生

---

### 3.3 维度 3：解决率（缩写：RR · Resolution Rate）

> **核心问题**：问题最终被有效解决的比例如何？是真懂还是虚假"懂了"？

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| RR-1 | 总解决率 | L1 | <30% | 50-70% | ≥85% |
| RR-2 | self_resolve 比例 | L1 | <20% 或 >90%（异常区） | 20-40% 或 70-90% | 40-70%（健康区） |
| RR-3 | 平均对话轮次 | L2 | ≥10 或 ≤2 | 7-9 或 3-4 | 5-7（合理脚手架长度） |
| RR-4 | 放弃率 | L1 | ≥40% | 20-30% | <10% |

#### RR-1：总解决率（自动）

- 公式：`resolved / total_questions`
- 直观但不能单独看（高解决率 + 极低 self_resolve 比例 = 老师强行宣布解决，要警惕）

#### RR-2：self_resolve 比例（自动；甜区指标）

- 公式：`resolution_sources["self_resolve"] / (resolution_sources["self_resolve"] + resolution_sources["teacher_marked"])`
- **不是越高越好**，是甜区分布：
  - **过高（>90%）**：学生过早 [懂了]，可能 prompt 滥用或老师给出答案就让学生草草收尾
  - **过低（<20%）**：老师完全靠 teacher_marked 强制结束，学生未真正消化
  - **甜区（40-70%）**：学生主动复述+真懂的迹象，最理想
- **0 分**：< 20% 或 > 90%
- **2 分**：20-40% 或 70-90%
- **4 分**：40-70%

#### RR-3：平均对话轮次（派生）

- 公式：`mean(len(d.history) for d in dialogs)` 单位 = 总消息条数（teacher + student 各算一条）
- **过短**（≤2）：很可能是"问题→答案"灌输，没有脚手架
- **过长**（≥10）：可能反复绕远未击中要点
- **甜区**（5-7）：合理脚手架，2-3 轮老师追问 + 学生复述

#### RR-4：放弃率（自动）

- 公式：`abandoned / total_questions`
- 越低越好

### 3.4 维度 4：师范生提问质量（缩写：TQ · Teacher Question Quality）

> **核心问题**：师范生作为操作者，在对话中表现出的教学语言能力如何？

> **此维度全部 LLM-Judge**（无法纯结构化）。LLM-Judge 拿对话 + 学生人设 + 教案元数据，输出此维度评分 + 1-2 句证据。

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| TQ-1 | 追问深度 | L3 | 重复学生原话 | 偶尔追问到错误前提 | 每次都引导学生说出错误前提 |
| TQ-2 | 解释分级 | L3 | 对所有学生说一样的话 | 部分适配人设 | 对优秀生跳步、对薄弱生分层 |
| TQ-3 | 维果茨基脚手架 | L3 | 直接告知答案 | 偶尔让学生举例复述 | 每次都让学生口头复述 / 自举反例 |
| TQ-4 | 错题诊断 | L3 | 只评对错 | 偶有诊断 | 每次准确指出根因（迷思 / 跳步 / 知识漏） |
| TQ-5 | 学段语言适配 | L3 | 术语过深或过简 | 部分适配 | 完全用学段词汇 |

#### TQ-1：追问深度

- 看老师是否善于"问回去"而不是"答下去"
- **4 分**例：学生说"力越大物体跑越快" → 老师问"那匀速运动的物体上有没有力？"（精准追问到错误前提）
- **0 分**例：学生说"力越大物体跑越快" → 老师说"对的，力可以让物体加速"（错误强化）

#### TQ-2：解释分级

- 看老师是否根据学生人设（`subject_level / speech_style / behavior_traits`）调整解释深度
- **4 分**例：对 `xueba` 类学生跳步用"由对称性可知"，对 `lost` 类同一问题"我们先看看左半边和右半边长得一样吗？"
- **0 分**例：所有学生收到同一段标准答案

#### TQ-3：维果茨基脚手架

- 看老师是否在学生说"懂了"之前先让学生**口头复述 / 自举反例**确认真懂
- **4 分**例：老师讲完后说"你来用自己的话说一遍/你来举一个例子"，学生确实复述了
- **0 分**例：老师讲完直接进下一题，或学生敷衍 [懂了] 老师就放过

#### TQ-4：错题诊断

- 看老师是否在判错后**说出错的根因类别**（迷思 / 跳步 / 知识漏 / 计算粗心）
- **4 分**例："你这里不是不会算，是把 / 当成了 ÷ 的反向，这是个分数迷思"
- **0 分**例："这里错了再算一遍"

#### TQ-5：学段语言适配

- 学段共性层（`StageProfile`）已限定词汇与思维方式：
  - `p_lower` 不能用"代数式 / 通分"
  - `h` 可以用"由 IVT 知"
- 看老师是否触及学段红线
- **4 分**：始终用学段适配词汇
- **0 分**：明显跨学段（小学讲"洛必达"、高中只说"差不多就行"）

### 3.5 维度 5：学生满意度（缩写：SS · Student Satisfaction）

> **核心问题**：从虚拟学生的视角看，他被讲懂了吗？情绪上是否被尊重？

| # | 指标 | 数据源 | 0 分 | 2 分 | 4 分 |
|---|---|---|---|---|---|
| SS-1 | self_resolve 比例 | L1 | <20% 或 >90%（异常） | 20-40% 或 70-90% | 40-70% |
| SS-2 | 末轮情绪 | L3 | frustrated（沮丧） | neutral | calm + 主动总结 |
| SS-3 | 主动追问次数 | L2 | 0 次主动追问 | ≥1 次 / 对话 | ≥2 次 / 对话且追问质量高 |
| SS-4 | 接受度（非敷衍懂了） | L3 | 仅"嗯""[懂了]" | 偶有复述 | 每次复述+举例+细节追问 |

> SS-1 与 RR-2 共用同一原始数据，但维度视角不同：RR-2 是从老师作答效率角度，SS-1 是从学生消化角度。两者一致是必然。

#### SS-2：末轮情绪（LLM-Judge）

- LLM-Judge 看每条 resolved dialog 的最后一条学生消息，判断情绪标签：
  - `calm` — "原来这样，我之前一直以为是…，下次我会注意"
  - `neutral` — "[懂了]"
  - `frustrated` — "好吧…还是不太明白但算了"

- 维度得分 = 学生消息中 calm / total_resolved 的占比映射到 0-4

#### SS-3：主动追问次数

- 派生指标：学生 `role=student` 消息中含 `?` 或 `？` 且不是首条问题
- 主动追问反映学生**真听进去了 + 有动力深入**

#### SS-4：接受度（LLM-Judge）

- 与 SS-2 不同：SS-4 看学生**是否真消化**，而非情绪
- **4 分**：每次老师讲完，学生都做了"复述 / 举类似例子 / 追问推论"三件之一
- **0 分**：学生只 "嗯""[懂了]" 走流程

## 4. 综合得分与展示策略

### 4.1 不做单一总分

如第 1 节所述，v0.5 不强行加权汇总。每个维度独立呈现：

```
维度 1（MR · 迷思破除）：       2.8 / 4
维度 2（KC · 重点覆盖）：       3.5 / 4
维度 3（RR · 解决率）：          2.0 / 4   ← 警告：self_resolve 89% 偏高
维度 4（TQ · 提问质量）：        2.4 / 4
维度 5（SS · 学生满意度）：      3.0 / 4
```

### 4.2 维度内得分计算

每个维度内取**算术平均**（v0.5 不做指标加权）：

```
维度得分 = mean([每个指标的 0-4 分]) 保留 1 位小数
```

### 4.3 反馈页可视化建议（B-Full 参考）

- 五边形雷达图（5 维度各占一角）
- 每个维度卡片下展开 3-5 个指标 + 数据来源标签（L1/L2/L3 用不同颜色区分）
- 红线告警：当某指标处于 0-1 分区间，前端高亮 + 给一句改进建议（来自第 5 节 LLM-Judge 输出的 `improvement_hint` 字段）

### 4.4 比较口径

- v0.5 不做"老师 A vs 老师 B"的横向比较；M3 评估 Agent 落地后再讨论
- v0.5 只做"同一师范生本节课各维度强弱对比"——发现自己的薄弱维度并迭代

### 5.2 LLM-Judge System Prompt（种子版）

```
你是一位资深的中小学教研员，擅长评估师范生的 1v1 答疑陪练表现。

打分原则：
1. 每个指标按 0/1/2/3/4 整数打分，不允许小数
2. 每个评分必须给出 1-2 句话的"证据片段"（直接引用对话原文 + 简短解释）
3. 仅评估师范生的表现，不评学生像不像（学生像不像由 prompt 工程负责）
4. 如证据不足以判断某个指标，标 N/A 而不是中庸 2 分

输出严格遵循以下 JSON schema，禁止额外字段、禁止 markdown、禁止解释性前后缀：

{
  "MR": {
    "MR_3": {"score": 0-4 | "N/A", "evidence": "..."},
    ... 仅 MR-3（其他 MR 由本地计算）
  },
  "TQ": {
    "TQ_1": {"score": ..., "evidence": "..."},
    "TQ_2": ...,
    "TQ_3": ...,
    "TQ_4": ...,
    "TQ_5": ...
  },
  "SS": {
    "SS_2": {"score": ..., "evidence": "..."},
    "SS_4": ...
  },
  "improvement_hints": [
    "对薄弱生 X，下次建议先用 ... 类比",
    "迷思 Y 你识别到了但解释跳得快，可以用 ..."
  ]
}
```

### 5.3 调用约定

- 每节课调一次 LLM-Judge（单次评估传入完整对话）
- 模型选择与生成模型不同（默认评估用 `deepseek-chat`，生成用 `ecnu-max`）
- temperature ≤ 0.3 保证可复现
- 失败重试 2 次后报错，不做静默兜底

## 7. 已知局限与 v1.0 改进方向

### 7.1 v0.5 已知局限

| 局限 | 影响 | v1.0 计划 |
|---|---|---|
| 五维度等权 | 但实际"迷思破除"应权重更高 | 加入维度权重，从用户测试反推合理值 |
| LLM-Judge 单次调用 | 长对话可能超 context | M2 末期实测；超长则按 dialog 分块评 |
| 无"跑题转化"专项指标 | 样本 C 展示了重要能力但未独立打分 | v1.0 在 TQ 维度加 TQ-6 |
| 缺少老师"自我反思"维度 | 当前只评对话 | M3 加入 self_reflection 阶段 + 评分 |
| 跨节课纵向追踪缺失 | 同一师范生的进步轨迹无法呈现 | M3 加历史评估表 + 个人成长曲线 |
| LLM-Judge 与生成模型同源时偏差 | 默认 ecnu-max | v1.0 强制要求异源（评估用 deepseek） |
